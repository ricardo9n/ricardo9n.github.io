Tutorial – Publicando um site no GitHub Pages (iniciante)

Parte 0 – Visão geral e preparo

Antes de começar, vamos alinhar expectativas e garantir que você está no caminho certo.

O que você vai conseguir ao final

Ao terminar este tutorial, você será capaz de:

  • Criar uma conta no GitHub
  • Enviar os arquivos do seu site (HTML, CSS e JavaScript)
  • Publicar esse site gratuitamente usando o GitHub Pages
  • Acessar seu site por um link público na internet
  • Atualizar o site sempre que quiser

Tudo isso sem precisar saber Git, versionamento ou usar terminal.

O que é GitHub Pages (em palavras simples)

O GitHub Pages é um serviço gratuito do GitHub que permite transformar uma pasta com arquivos de site (HTML, CSS e JS) em um site acessível pela internet.

Pense assim:

  • Seu computador: você abre o index.html clicando duas vezes
  • GitHub Pages: qualquer pessoa abre esse mesmo site por um link

No início do vídeo, a autora reforça exatamente isso: você não precisa entender nada de hospedagem, Git ou servidores.

Vídeo (introdução e tranquilização do iniciante): https://www.youtube.com/watch?v=q3vsAtijvgA&t=0s

O que você precisa ter pronto antes de seguir

Antes de continuar, confira se você já tem:

  • Uma pasta no seu computador com os arquivos do site
  • Um arquivo chamado index.html
  • Opcionalmente, arquivos .css e .js

Exemplo simples de estrutura correta:

  • index.html
  • style.css
  • script.js

Atenção: o arquivo index.html é obrigatório. Ele é a “porta de entrada” do site.

Se isso estiver ok, podemos seguir.

Parte 1 – Criando sua conta no GitHub

Agora vamos criar sua conta no GitHub. Esse será o lugar onde seu site ficará armazenado.

Passo 1 – Acessar o site do GitHub

Abra o navegador e acesse:

https://github.com

Você verá a página inicial do GitHub.

Passo 2 – Criar uma conta (Sign up)

Se você ainda não tem conta:

  • Clique em Sign up

  • Preencha:

    • E-mail
    • Senha
    • Nome de usuário

  • Siga o passo a passo até concluir o cadastro

No vídeo, esse momento aparece logo após a introdução, quando a autora diz que quem é novo pode simplesmente clicar em “Sign up” e preencher os dados.

Vídeo (criação da conta): https://www.youtube.com/watch?v=q3vsAtijvgA&t=70s

Passo 3 – Fazer login (Sign in)

Depois de criar a conta:

  • Volte para https://github.com
  • Clique em Sign in
  • Entre com seu usuário e senha

Ao entrar, você verá a página inicial da sua conta.

O que você está vendo agora (visão rápida)

Não se preocupe em entender tudo agora. Para este tutorial, você só precisa reconhecer:

  • A barra superior
  • O botão + (para criar coisas novas)
  • A opção New repository

Na próxima parte, vamos usar exatamente isso para criar o espaço onde seu site vai ficar.

Pare aqui por um momento. Quando estiver com a conta criada e logado no GitHub, siga para a Parte 2.

Parte 2 – Criando o repositório do site

Agora vamos criar o repositório, que é o local onde os arquivos do seu site vão ficar guardados dentro do GitHub.

O que é um repositório (sem complicação)

Um repositório é, basicamente, uma pasta na internet.

Em vez de estar no seu computador, essa pasta fica no GitHub.

Dentro dela, você vai colocar:

  • index.html
  • arquivos .css
  • arquivos .js
  • imagens (se tiver)

No vídeo, a autora explica isso de forma simples, dizendo que o repositório é apenas o lugar onde o código fica guardado.

Vídeo (explicação e início da criação do repositório): https://www.youtube.com/watch?v=q3vsAtijvgA&t=95s

Passo 1 – Iniciar a criação do repositório

Existem duas formas equivalentes de criar um repositório no GitHub:

  • Clicando no botão New
  • Ou clicando no botão + no canto superior direito e depois em New repository

No vídeo, a autora mostra as duas opções e diz que você pode usar qualquer uma delas.

Vídeo (onde clicar para criar): https://www.youtube.com/watch?v=q3vsAtijvgA&t=114s

Clique em uma dessas opções para continuar.

Passo 2 – Escolher o nome do repositório

Agora você precisa dar um nome para o repositório.

Esse nome:

  • Pode ser simples
  • Não precisa ser técnico
  • Pode representar o site (ex.: portfólio, landing page, site pessoal)

Exemplos de nomes válidos:

  • meu-site
  • portfolio
  • landing-page

No vídeo, a autora usa como exemplo um nome relacionado a uma landing page.

Vídeo (nomeando o repositório): https://www.youtube.com/watch?v=q3vsAtijvgA&t=121s

Dica: evite espaços e letras maiúsculas. Use letras minúsculas e hífen - se quiser separar palavras.

Passo 3 – Escolher a visibilidade (Public ou Private)

Aqui existe um ponto muito importante.

Você verá duas opções:

  • Public
  • Private

Para usar o GitHub Pages gratuitamente, o repositório precisa ser Public.

Public significa:

  • Qualquer pessoa pode ver os arquivos
  • Seu site poderá ser acessado pela internet

No vídeo, a autora marca explicitamente a opção Public antes de criar o repositório.

Vídeo (escolha da visibilidade): https://www.youtube.com/watch?v=q3vsAtijvgA&t=132s

Atenção: se você escolher Private, o site não será publicado gratuitamente.

Passo 4 – Criar o repositório

Depois de:

  • escolher o nome
  • marcar como Public

Clique no botão Create repository.

Pronto. O repositório foi criado.

Você será levado para uma nova página, que por enquanto estará vazia ou com poucas informações.

Na próxima parte, vamos enviar os arquivos do site para dentro desse repositório.

Siga para a Parte 3 quando estiver com o repositório criado.

Parte 3 – Enviando os arquivos do site para o repositório

Agora vamos colocar, dentro do repositório, os arquivos do seu site (HTML, CSS e JavaScript).

Este é um ponto muito importante do processo. Se algo der errado aqui, o site pode não abrir depois. Por isso, siga com atenção.

Antes de começar: confira sua pasta no computador

No seu computador, você deve ter uma pasta aberta contendo os arquivos do site.

Exemplo de estrutura correta:

  • index.html
  • style.css
  • script.js
  • imagens/

Atenção: o arquivo index.html deve estar diretamente dentro da pasta, e não dentro de outra subpasta.

Erro comum (muito comum mesmo)

Um erro frequente é subir a pasta inteira, em vez de subir os arquivos que estão dentro dela.

Errado:

  • pasta-do-site

    • index.html
    • style.css

Certo:

  • index.html
  • style.css

No vídeo, a autora faz questão de alertar sobre isso e mostra exatamente esse cuidado.

Vídeo (alerta sobre não subir a pasta errada): https://www.youtube.com/watch?v=q3vsAtijvgA&t=142s

Passo 1 – Iniciar o upload dos arquivos

No repositório que você acabou de criar, procure pela opção:

Uploading an existing file

Clique nela.

Essa opção permite enviar arquivos do seu computador diretamente pelo navegador, sem usar nenhum programa extra.

Vídeo (onde clicar para fazer upload): https://www.youtube.com/watch?v=q3vsAtijvgA&t=141s

Passo 2 – Arrastar os arquivos corretos

Com a tela de upload aberta:

  • Abra a pasta do seu site no computador
  • Entre na pasta (não arraste a pasta fechada)
  • Selecione os arquivos que estão dentro dela
  • Arraste esses arquivos para a área indicada no GitHub

Você verá os arquivos aparecendo na tela.

Aguarde até que todos sejam carregados.

Passo 3 – Confirmar o envio (Commit)

Depois que os arquivos aparecerem na tela, desça a página.

Você verá um campo para escrever uma mensagem.

Pode escrever algo simples, por exemplo:

  • "Primeira versão do site"
  • "Adicionando arquivos do site"

Em seguida, clique no botão Commit changes.

No vídeo, a autora escreve uma mensagem simples e confirma o envio exatamente assim.

Vídeo (commit dos arquivos): https://www.youtube.com/watch?v=q3vsAtijvgA&t=187s

O que aconteceu agora

Neste momento:

  • Seus arquivos já estão no GitHub
  • O repositório deixou de estar vazio
  • O site ainda não está no ar (isso vem na próxima parte)

Se você voltar para a página principal do repositório, já deverá ver o index.html listado.

Se não aparecer, algo deu errado no upload — volte e revise os passos.

Na próxima parte, vamos ativar o GitHub Pages e colocar o site no ar.

Siga para a Parte 4 quando seus arquivos já estiverem visíveis no repositório.

Parte 4 – Ativando o GitHub Pages e publicando o site

Agora vamos fazer o passo que realmente coloca o site no ar: ativar o GitHub Pages.

Até aqui, seus arquivos estão guardados no GitHub, mas ainda não estão acessíveis como um site.

Passo 1 – Acessar as configurações do repositório

Na página do seu repositório, procure no menu superior pela opção Settings.

Clique em Settings.

No vídeo, a autora mostra claramente esse momento antes de falar sobre GitHub Pages.

Vídeo (acessando Settings): https://www.youtube.com/watch?v=q3vsAtijvgA&t=210s

Passo 2 – Encontrar a seção Pages

Dentro das configurações:

  • Role a página para baixo
  • Procure pela opção Pages no menu lateral

Clique em Pages.

É aqui que o GitHub permite transformar o repositório em um site.

Vídeo (indo até Pages): https://www.youtube.com/watch?v=q3vsAtijvgA&t=216s

Passo 3 – Escolher a branch correta

Você verá uma opção chamada Branch.

Ela define de onde o GitHub vai pegar os arquivos do site.

Para este tutorial:

  • Escolha main
  • Na pasta, mantenha /(root)

Você não precisa entender o que é branch agora. O importante é usar:

  • main
  • root

No vídeo, a autora explica que o projeto só tem essa branch e por isso ela deve ser usada.

Vídeo (escolhendo branch e pasta): https://www.youtube.com/watch?v=q3vsAtijvgA&t=238s

Passo 4 – Salvar a configuração

Depois de selecionar:

  • Branch: main
  • Pasta: /(root)

Clique em Save.

A partir desse momento, o GitHub começa a trabalhar para publicar o site.

Atenção – tempo de espera

O site não aparece instantaneamente.

Normalmente:

  • Pode levar de 1 a 5 minutos
  • Em projetos maiores, pode demorar um pouco mais

No vídeo, a autora comenta que geralmente nunca passa de alguns minutos.

Vídeo (tempo de publicação): https://www.youtube.com/watch?v=q3vsAtijvgA&t=254s

Não feche a página. Aguarde um pouco.

O que acontece enquanto você espera

Enquanto você aguarda:

  • O GitHub verifica os arquivos
  • Organiza as pastas
  • Prepara o site para acesso público

Isso tudo acontece automaticamente.

Na próxima parte, vamos acessar a URL do site, testar no navegador e no celular.

Siga para a Parte 5 após alguns minutos de espera.

Parte 5 – Acessando a URL do site e testando

Depois de alguns minutos, o GitHub Pages finaliza a publicação do site. Agora vamos acessar o site pela internet e conferir se está tudo funcionando.

Passo 1 – Encontrar a URL do site

Volte para a seção Settings → Pages do seu repositório.

Quando o site estiver pronto, você verá uma mensagem informando que ele foi publicado, junto com um link parecido com este:

https://seu-usuario.github.io/nome-do-repositorio/

Essa é a URL pública do seu site.

No vídeo, a autora mostra exatamente esse momento, quando o link aparece e ela clica nele.

Vídeo (URL do site aparece): https://www.youtube.com/watch?v=q3vsAtijvgA&t=259s

Passo 2 – Abrir o site no navegador

Clique na URL exibida.

Seu site deve abrir normalmente, mostrando o conteúdo do index.html.

Se o site abrir corretamente, parabéns: ele já está no ar.

Passo 3 – Testar o site no celular

Um dos benefícios do GitHub Pages é que o site pode ser acessado de qualquer lugar.

Você pode:

  • Copiar a URL
  • Enviar para você mesmo por WhatsApp ou e-mail
  • Abrir no celular

No vídeo, a autora comenta que testou o site no celular e conseguiu navegar normalmente.

Vídeo (teste no celular): https://www.youtube.com/watch?v=q3vsAtijvgA&t=299s

O que fazer se o site não abrir imediatamente

Se ao clicar no link o site não abrir:

  • Aguarde mais 1 ou 2 minutos
  • Atualize a página (F5)
  • Volte em Settings → Pages e confira se a mensagem de publicação aparece

Isso é normal, especialmente na primeira vez.

Como usar esse link na prática

Agora que o site está no ar, você pode:

  • Colocar o link no currículo
  • Usar como portfólio
  • Enviar para entrevistas de emprego
  • Compartilhar com amigos ou clientes

No vídeo, a autora sugere exatamente esse uso para portfólio e entrevistas.

Vídeo (uso do site como portfólio): https://www.youtube.com/watch?v=q3vsAtijvgA&t=302s

Na próxima parte, vamos ver como atualizar o site depois, sem precisar refazer tudo.

Siga para a Parte 6 quando quiser alterar o conteúdo do site.

Parte 6 – Atualizando o site após alterações

Depois que o site está no ar, é muito comum querer corrigir textos, trocar imagens ou melhorar o visual.

A boa notícia é que não é preciso configurar tudo de novo. Atualizar o site é bem simples.

O que significa “atualizar o site” no GitHub Pages

Atualizar o site significa:

  • Alterar arquivos no seu computador (HTML, CSS, JS)
  • Enviar esses arquivos atualizados para o repositório

Sempre que você envia uma nova versão dos arquivos, o GitHub Pages atualiza o site automaticamente.

Passo 1 – Fazer alterações no seu computador

No seu computador:

  • Abra o arquivo que deseja alterar (por exemplo, index.html)
  • Faça as mudanças normalmente
  • Salve o arquivo

Você pode, por exemplo:

  • Corrigir um texto
  • Trocar uma cor no CSS
  • Alterar um botão

Passo 2 – Enviar os arquivos atualizados para o GitHub

Volte ao repositório no GitHub.

Você pode atualizar os arquivos de duas formas simples:

Opção 1 – Upload novamente pelo navegador

  • Clique em Add file
  • Escolha Upload files
  • Arraste os arquivos atualizados
  • Se o arquivo tiver o mesmo nome, ele será substituído

Opção 2 – Editar diretamente no GitHub (para mudanças pequenas)

  • Clique em um arquivo (por exemplo, index.html)
  • Clique no ícone de lápis (Edit)
  • Faça a alteração
  • Salve

Para iniciantes, a Opção 1 costuma ser a mais segura.

Passo 3 – Confirmar as alterações (Commit)

Assim como antes:

  • Escreva uma mensagem simples (ex.: "Atualizando conteúdo do site")
  • Clique em Commit changes

Depois disso, o GitHub Pages começa a atualizar o site.

Tempo para a atualização aparecer

Normalmente:

  • As mudanças aparecem em 1 ou 2 minutos
  • Às vezes é quase imediato

Se não aparecer:

  • Atualize a página do site (F5)
  • Aguarde mais um pouco

Você não precisa voltar em Settings ou Pages.

Importante lembrar

  • O link do site continua o mesmo
  • Você pode atualizar quantas vezes quiser
  • Não há custo nenhum

Isso torna o GitHub Pages ótimo para portfólios e sites simples.

Parte 7 – Erros comuns e como resolver

Mesmo seguindo o passo a passo, é normal algo não funcionar de primeira. Nesta parte, vamos ver os problemas mais comuns e como resolvê-los sem desespero.

Erro 1 – O site não abre (página não encontrada)

Sintoma:

  • Ao acessar a URL, aparece erro 404 ou página não encontrada

Possíveis causas e soluções:

  • Aguarde alguns minutos: a publicação pode ainda não ter terminado
  • Volte em Settings → Pages e confirme se o link aparece
  • Verifique se o repositório está como Public
  • Confirme se a branch está como main e a pasta como /(root)

Esse atraso inicial é normal, como o vídeo comenta durante a publicação.

Erro 2 – Página em branco

Sintoma:

  • O site abre, mas aparece tudo em branco

Causa mais comum:

  • Não existe um arquivo index.html no local correto

Como resolver:

  • Confirme que o arquivo se chama exatamente index.html
  • Verifique se ele está na raiz do repositório (e não dentro de outra pasta)

No vídeo, a autora reforça que o index.html é obrigatório para o site funcionar.

Vídeo (importância do index.html): https://www.youtube.com/watch?v=q3vsAtijvgA&t=150s

Erro 3 – CSS ou JavaScript não funcionam

Sintoma:

  • O HTML aparece, mas sem estilo
  • Botões não funcionam

Causa mais comum:

  • Caminho dos arquivos está errado

Exemplo de erro comum:

  • No HTML: href="/style.css"
  • No GitHub Pages, isso pode não funcionar

Forma mais segura:

  • Use caminhos relativos: href="style.css"
  • Confira se o nome do arquivo está exatamente igual (maiúsculas contam)

Erro 4 – Atualizei o site, mas não mudou

Sintoma:

  • Você fez alterações, mas o site continua igual

Soluções:

  • Aguarde 1 ou 2 minutos
  • Atualize a página do navegador (F5)
  • Limpe o cache do navegador ou teste em aba anônima

Isso é comum por causa do cache.

Erro 5 – Subi a pasta errada

Sintoma:

  • O site não carrega corretamente

Causa:

  • Os arquivos ficaram dentro de uma subpasta

Solução:

  • Entre no repositório
  • Apague a pasta errada
  • Suba novamente apenas os arquivos (index.html, css, js)

Esse é um dos erros mais comuns para quem está começando.

Quando tudo falhar

Se nada funcionar:

  • Revise cada parte do tutorial com calma
  • Confira especialmente a Parte 3 (upload dos arquivos)
  • Verifique novamente Settings → Pages

Quase sempre o problema está em:

  • nome do arquivo
  • pasta errada
  • caminho de CSS ou JS

Conclusão

Se você chegou até aqui, você já sabe:

  • Criar conta no GitHub
  • Criar um repositório público
  • Subir arquivos HTML, CSS e JS
  • Publicar um site no GitHub Pages
  • Atualizar o site
  • Resolver os erros mais comuns

Você agora tem um site no ar, gratuito e acessível de qualquer lugar.

A partir daqui, o próximo passo natural é melhorar o conteúdo, o visual e usar esse site como portfólio.
